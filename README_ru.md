**[New The Unarchiver](https://github.com/new-the-unarchiver)** — удобный и мощный инструмент для распаковки архивов. Его движок написан на Rust, а поверх него живут адаптеры для разных языков. Здесь находится пакет `newtua` — для языка **Python**.

Пакет только читает и распаковывает содержимое, он **не создаёт** архивы.

Сейчас ядро проекта поддерживает 55 форматов — от zip, 7z, rar и tar до образов дисков (DMG, ISO, WIM) и старых форматов вроде StuffIt и ARJ. Впоследствии их может стать больше.

Вся работа происходит внутри Python-кода, без запуска внешних оболочек или дополнительных процессов.

## Почему [New The Unarchiver](https://github.com/new-the-unarchiver)

- **Формат-agnostic.** Один и тот же код открывает любой из 55 форматов. Формат определяется по содержимому, а не по расширению.
- **Кросс-платформенность.** Ограничена только особенностями самих платформ. Тем не менее, различия минимизированы, насколько это было возможно.
- **Быстрый.** Движок на Rust, распаковка целиком в общем процессе, без подпроцессов и без промежуточных файлов там, где они не нужны.
- **Параллелизм из коробки.** Распаковка не блокирует GIL, поэтому несколько архивов могут быть распакованы по-настоящему параллельно. Мы также предоставляем готовые инструменты для потоков, процессов, `asyncio` и даже `gevent`.
- **Оптимизирован для питонистов.** Идиоматичный API: архив — это последовательность, запись — файловый объект, ошибки — типизированные исключения, а не коды возврата.
- **Типизированный.** Пакет помечен `py.typed` и проходит `mypy --strict` — типы подхватываются в IDE, а ваш CI/CD не спотыкается.

## Установка

```bash
uv add newtua
```

Нужен Python 3.11 или новее. Пакет распространяется готовыми колёсами `abi3` (одно колесо на платформу покрывает все версии Python ≥ 3.11), так что установка не требует ни Rust, ни компилятора.

## Быстрый старт

```python
import newtua

# Напечатать содержимое архива и распаковать его в папку `out/`
with newtua.Archive("photos.zip") as ar:
    for entry in ar:
        print(entry.path, entry.size)
    ar.extract("out/")

# Прочитать одну запись целиком в память
with newtua.Archive("archive.7z") as ar:
    data = ar.read("readme.txt")

# Распаковать пачку архивов параллельно, по ядрам, в разные папки
results = newtua.extract_many(
    [("a.zip", "out/a"), ("b.7z", "out/b")],
    on_result=lambda job, report: print(job.archive, report.extracted),
)
```

## `Archive` — синхронный доступ к одному архиву


Основной класс. Используется для чтения и распаковки архивов.

```python
newtua.Archive(source, *, password=None, encoding=None)
```

`source` — путь к файлу (`str` или `os.PathLike`), готовые `bytes` или уже открытый двоичный файловый объект. Конструктор ничего не читает сразу: настоящее открытие происходит при первом обращении к архиву. Это сделано специально — так можно сначала проверить, нужен ли пароль, и только потом его установить. Если пароль известен заранее, можно передать его сразу в конструктор.

```python
import pathlib
import newtua

# Инициализация архива по пути на диске
with newtua.Archive("archive.7z") as ar:
    print(len(ar))

# Инициализация архива из байтов, без временного файла на вашей стороне
data = pathlib.Path("archive.7z").read_bytes()
with newtua.Archive(data) as ar:
    print(len(ar))

# Инициализация архива из уже открытого файлового объекта
with open("archive.7z", "rb") as f, newtua.Archive(f) as ar:
    print(len(ar))
```

`Archive` работает и без `with`, но тогда его нужно закрывать вручную через метод `ar.close()`. Он удалит временный файл (если источником были байты или поток) и освободит занятые ресурсы.

### Свойства Archive

| Свойство | Тип | Что это |
|---|---|---|
| `format` | `newtua.Format` | Формат архива, например `"7z"` или `newtua.Format.RAR`. Внутри это `StrEnum`, со всеми его возможностями. |
| `needs_password` | `bool` | Нужен ли пароль, чтобы прочитать содержимое. Можно проверить до того, как спрашивать пароль у пользователя. |
| `password` | `str \| None` | Текущий пароль. Присваивание открывает архив заново с новым паролем. |
| `detected_encoding` | `str` | Кодировка, которую движок автоматически определил для имён файлов (например `"windows-1251"`). |

### Последовательность содержимого архива


> Здесь и далее, когда речь идёт о содержимом архива, мы используем термин «запись» вместо «файл», потому что это также могут быть директории и символические ссылки.



`Archive` ведёт себя как обычная Python-последовательность (Sequence):

```python
with newtua.Archive("archive.7z") as ar:
    len(ar)                 # число записей
    for entry in ar: ...    # перебрать записи
    ar[0]                   # запись по индексу
    ar["a.txt"]             # запись по имени (первая с таким именем — оно может повторяться)
    "a.txt" in ar           # есть ли такая запись
    ar[0:10]                # срез — кортеж записей
```

#### Исключения

- Обращение по несуществующему имени вызывает `EntryNotFoundError` (это подкласс `KeyError`, так как код типа `ar["имя"]` ведёт себя как обращение к словарю).
- Числовой индекс вне диапазона — `IndexError` (стандартно для `Sequence`).
- Обращение к уже закрытому архиву — `ValueError` (как для закрытого файла).

### Методы Archive

```python
ar.read(entry) -> bytes
```
Прочитать одну запись целиком в память. `entry` — номер, имя, путь или сама `Entry`.

```python
ar.open(entry, *, stream=False) -> EntryStream
```
Открыть запись как файловый объект (см. [EntryStream](#entrystream--файловый-объект-записи)).

```python
ar.extract(
    dest, *, selection=None, wrapper=True, strict=False, preserve=True, progress=None
) -> newtua.Report
```
Распаковать записи в папку `dest`. Подробнее о параметрах метода `extract` и возвращаемом им значении см. ниже.

```python
ar.close()
```
Закрыть архив и удалить временный файл, если он создавался.

### Распаковка

```python
with newtua.Archive("archive.7z") as ar:
    report = ar.extract("out/")
    print(report.extracted, report.failed, report.aborted)
```

`extract` возвращает `newtua.Report` — число распакованных (`extracted`) и не распакованных (`failed`) записей и флаг отмены (`aborted`).

Параметры метода `extract`:

- `selection` — список номеров, имён или самих записей: распаковать только их.
- `wrapper` (по умолчанию `True`) — содержимое без общего корневого каталога разворачивается в папку, названную по имени архива (как было в оригинальном The Unarchiver). `wrapper=False` — распаковать как есть. Имя обёртки берётся из имени источника; у архива из голых байтов или `io.BytesIO` имени нет — тогда обёртки не будет.
- `strict` — прерывать распаковку на первой же небезопасной записи (путь, выходящий за пределы `dest`), а не пропускать её.
- `preserve` — восстанавливать права и время файлов, где они записаны.
- `progress` — обработчик прогресса (см. [Прогресс распаковки](#прогресс-распаковки)).

### Пароль

`needs_password` подсказывает, зашифрован ли архив. Пароль передаётся либо сразу в конструкторе, либо позже через свойство `password` — смена пароля просто открывает архив заново.

```python
ar = newtua.Archive("secret.zip")
if ar.needs_password:
    ar.password = "секрет"
data = ar.read(0)
ar.close()
```

Отсутствие пароля там, где он нужен, вызывает `PasswordRequiredError`, неверный пароль — `WrongPasswordError`.

### Кодировка имён

Имена в старых архивах не всегда в UTF-8. `Archive` сам определяет кодировку — один вердикт на весь архив, разом по настоящим байтам всех имён:

```python
with newtua.Archive("старый.zip") as ar:
    print(ar.detected_encoding)          # "windows-1251"
    for entry in ar:
        print(entry.raw_name.decode(ar.detected_encoding))
```

Параметр `encoding=` в конструкторе переопределяет автоопределение, если оно ошиблось; `detected_encoding` тогда возвращает присвоенное значение.

### Формат архива

```python
with newtua.Archive("archive.7z") as ar:
    print(ar.format)                           # "7z"
    print(ar.format == newtua.Format.RAR)      # False
    print(ar.format == "7z")                   # True — Format это StrEnum
```

### Прогресс распаковки

`extract(..., progress=...)` вызывает обработчик по одному событию за раз:

```python
def show(event: newtua.ProgressEvent) -> None:
    match event:
        case newtua.EntryStarted(path=path, size=size):
            print(f"{path} ({size} байт)")
        case newtua.BytesWritten(written=written):
            pass  # обновляем прогресс-бар или другую индикацию
        case newtua.EntryFinished():
            print("готово")

with newtua.Archive("big.zip") as ar:
    ar.extract("out/", progress=show)
```

События — `EntryStarted`, `BytesWritten`, `EntryFinished` (все наследуют `ProgressEvent`).

Верните из обработчика `False`, чтобы отменить распаковку на середине: она прервётся на текущей записи, а остальные будут пропущены.

## `Entry` — одна запись архива

`Entry` — это неизменяемые метаданные одной записи и заодно удобные методы, чтобы прочитать или распаковать только её одну. Получить `Entry` можно при переборе архива или по индексу/имени/пути.

### Поля

| Поле | Тип | Что это |
|---|---|---|
| `index` | `int` | Позиция в архиве. Именно она однозначно определяет запись: имена могут повторяться. |
| `path` | `path.PurePosixPath` | Декодированный и нормализованный путь — с ним и работают. |
| `raw_name` | `bytes` | Имя ровно теми байтами, какими оно записано в архиве, без расшифровки. |
| `kind` | `newtua.EntryKind` | `FILE`, `DIR` или `SYMLINK` (внутри это `StrEnum`). |
| `size` | `int` | Размер в байтах. |
| `is_encrypted` | `bool` | Зашифрована ли запись. |
| `mode` | `int \| None` | Права доступа Unix, если применимо. |
| `mtime` | `datetime \| None` | Время изменения, если записано. |

`raw_name` — это то, на что должны смотреть проверки безопасности пути: декодирование могло бы исказить имя, ведь в архиве оно не обязано быть в UTF-8. Декодировать его самому можно через `entry.raw_name.decode(ar.detected_encoding)`.

### Методы Entry


```python
entry.is_file() -> bool
entry.is_dir() -> bool
entry.is_symlink() -> bool
```
Проверить тип записи.

```python
entry.read() -> bytes
```
Прочитать эту запись целиком (аналог `ar.read(entry)`). Метод `read()` возвращает байты, а не файловый объект, как `open()`.

```python
entry.open(*, stream=False) -> newtua.EntryStream
```
Открыть как файловый объект (см. [EntryStream](#entrystream--файловый-объект-записи)).

```python
entry.extract(dest) -> newtua.Report
```
Распаковать только эту запись в `dest`, **без** папки-обёртки (т.к. запись всего одна). Собственный путь записи внутри архива при этом сохраняется.

> **Оговорка про асинхронность.** У записей, полученных из `AsyncArchive`, доступны только метаданные. Методы `read()`, `open()` и `extract()` у них не работают — вызов любого поднимет `ValueError`. Читайте такие записи через сам `AsyncArchive` (`await ar.read(entry)`), чтобы не запустить блокирующую операцию прямо в цикле событий. Подробно см. [Параллелизм и асинхронность](#параллелизм-и-асинхронность)

## `EntryStream` — файловый объект записи

`ar.open(...)` возвращает `EntryStream` — обычный двоичный файловый объект (наследник `io.BufferedIOBase`). Его можно передать в `shutil.copyfileobj`, или в `json.load`, или обернуть в `io.TextIOWrapper`, чтобы читать текстовую запись построчно:

```python
import io, newtua

with newtua.Archive("archive.7z") as ar, ar.open("a.txt") as f:
    for line in io.TextIOWrapper(f, encoding="utf-8"):
        print(line)
```

> **Оговорка про типы.** `EntryStream` наследует `io.BufferedIOBase`, а не протокол `typing.BinaryIO`. На аннотацию через `BinaryIO` (или `IO[bytes]`) `mypy` выдаст ошибку, хотя сам код будет рабочим. Свои функции под этот объект аннотируйте как `io.BufferedIOBase` или прямо `newtua.EntryStream`.

### Чтение на лету (`stream=True`)

У чтения два режима, обычный и на лету, по умолчанию — обычный.

**Обычный (`stream=False`).** Запись распаковывается целиком заранее: небольшая остаётся в памяти, крупная уходит во временный файл. Результат перематывается (`seek`, `tell`) — этого ждёт большинство кода.

**На лету (`stream=True`).** Запись распаковывается рабочим потоком прямо в канал ОС (pipe), а вы читаете её с другого конца.

Три отличия от обычного режима:

- **Память постоянная** — независимо от размера записи, хоть 100 гигабайт.
- **Первый байт приходит сразу**, не дожидаясь распаковки последнего.
- **Перемотки нет** — `seekable()` даёт `False`, а `seek()`/`tell()` приводят к `io.UnsupportedOperation`.

Пример: считаем хеш SHA-256 для большой записи на лету.

```python
import hashlib, newtua

with newtua.Archive("huge.zip") as ar, ar.open("big.iso", stream=True) as f:
    digest = hashlib.sha256()
    while chunk := f.read(1024 * 1024):
        digest.update(chunk)
    print(digest.hexdigest())
```

Берите этот режим для большой записи, по которой нужен всего один проход: посчитать хеш, перелить в сеть, разобрать потоковым парсером. Для всего остального удобнее обычный режим.

Ещё две удобные особенности этого режима:

- **Поток не ломается, если закрыть архив через `close()`.** Вызов `open(..., stream=True)` не завершается мгновенно: он ждёт, пока рабочий поток откроет архив, и только тогда отдаёт вам поток для чтения. К этому моменту рабочий поток уже держит архив открытым сам — поэтому вы можете сразу закрыть `Archive` через `ar.close()` и спокойно дочитать поток до конца, чтению это не помешает.
- **Ошибку открытия видно сразу.** Если архив не удаётся открыть — неверный пароль, битый заголовок, — `open(..., stream=True)` поднимет исключение прямо на этом вызове. Вы не окажетесь в ситуации, когда поток вроде бы открылся, чтение пошло, а на середине оборвалось без объяснения.

**Под Windows** режим работает только для источника-**пути**: Rust сам создаёт канал, а Python читает из него. Для источника-`bytes` или потока будет `NotImplementedError` (см. [Ограничения под Windows](#ограничения-под-windows)).

## Параллелизм и асинхронность

Мы специально написали Rust-слой так, чтобы распаковка отпускала GIL, — тогда она может идти по-настоящему параллельно.

Мы также сделали два готовых набора инструментов, чтобы разработчикам не пришлось самим возиться с потоками, каналами и передачей объектов между ними.

### Несколько архивов: `extract_many` / `list_many`

```python
newtua.extract_many(
    jobs, *, backend=newtua.Backend.THREAD, max_workers=None, on_result=None, on_error=None, cancel=None,
) -> list[newtua.BatchResult]
```

Метод `extract_many` параллельно распаковывает несколько архивов. Здесь `jobs` — либо кортежи `(архив, куда)`, либо объекты `newtua.ExtractJob` для более тонкой настройки:

```python
import newtua

jobs = [
    ("a.zip", "out/a"),
    newtua.ExtractJob(archive="b.7z", dest="out/b", password="секрет"),
]
results = newtua.extract_many(jobs, on_result=lambda job, r: print(job.archive, r.extracted))
for res in results:
    if res.error is None:
        print(res.job.archive, "→", res.report.extracted)
    else:
        print(res.job.archive, "ошибка:", res.error)
```

 ```python
 newtua.ExtractJob(archive, dest, *, password=None, encoding=None, selection=None, wrapper=True, strict=False, preserve=True, progress=None)
 ```
Одна задача со своими опциями. Параметры те же, что у [Archive.extract](#распаковка).

```python
newtua.BatchResult(job, report, error)
```
Результат одной задачи: содержит либо `report`, либо `error`. Распаковка **никогда не падает на первой ошибке** — один битый архив не мешает распаковать остальные, а его исключение доступно в параметре `error`.

```python
newtua.list_many(
    archives, *, password=None, encoding=None, backend=newtua.Backend.THREAD, max_workers=None, on_result=None, on_error=None,
) -> list[newtua.ListingResult]
```

Метод `list_many` параллельно читает оглавления нескольких архивов. Здесь `archives` — список путей к архивам:

```python
for res in newtua.list_many(["a.zip", "b.7z", "c.rar"]):
    if res.error is None:
        print(res.archive, len(res.entries))
```

`newtua.ListingResult(archive, entries, error)` — содержит либо `entries`, либо `error`.

### `AsyncArchive` — архив в asyncio

```python
newtua.AsyncArchive(source, *, password=None, encoding=None)
```

Это тот же `Archive`, но вся работа с байтами здесь происходит вне цикла событий — таким образом, распаковка его не блокирует. Параметр `stream` в методе `.open` отсутствует.

```python
async with newtua.AsyncArchive("big.dmg") as ar:
    print(ar.format, len(ar))            # метаданные — синхронно, без await
    for entry in ar:                     # обход записей тоже синхронный
        print(entry.path, entry.size)

    data = await ar.read("readme.txt")           # чтение — вне цикла событий
    report = await ar.extract("out/")            # распаковка — вне цикла событий

    async with ar.open("payload.bin") as stream:  # потоковое чтение
        async for chunk in stream:
            ...
```

Как работает `AsyncArchive`:

1. **Метаданные читаются синхронно.** `format`, `detected_encoding`, `needs_password`, `len(ar)`, перебор, `ar[0]`, `ar["имя"]`, `"имя" in ar`, срезы — это чистый Python по кэшу, поэтому `await` здесь не нужен. Ждать нужно только реальную работу с байтами.
2. **`await ar.read(entry) -> bytes`** и **`await ar.extract(dest, *, selection=None, wrapper=True, strict=False, preserve=True, progress=None) -> Report`** — выглядят как у `Archive`, но вызываются с `await`.
3. **`ar.open(entry)`** — асинхронный контекст-менеджер, который отдаёт `AsyncEntryStream` (потоковое чтение, память постоянная). Использование: `await stream.read(n)`, `async for chunk in stream`, `await stream.aclose()`.
4. **Блокирующих методов нет.** Все блокирующие методы `Archive` либо заменены на их асинхронные аналоги, либо удалены. У `AsyncArchive` попросту нет синхронных `read`/`extract` — только их `await`-версии. Ошибку поймает тайпчекер, ещё до запуска.

### Потоки, процессы, gevent — возможности и ограничения

- **Потоки** (`Backend.THREAD`, и `asyncio` через `to_thread`) — самый простой вид параллелизма. Распаковка идёт в настоящих потоках не блокируя GIL. Правило: **один архив — один поток** (объект читателя привязан к своему потоку); зато независимых архивов в разных потоках может быть сколько угодно.
- **Процессы** (`Backend.PROCESS`) — полная изоляция: падение в одном процессе не роняет остальные. У этого бэкенда три особенности, о которых стоит знать заранее:
    - **Archive/AsyncArchive в процесс не передаётся.** Открытый `Archive`/`AsyncArchive` нельзя отправить в другой процесс — попытка сделать это приведёт к `TypeError`. Поэтому `extract_many` принимает **только пути** к архивам, а каждый архив открывается уже внутри своего воркера.
    - **`progress` недоступен.** Функцию-колбэк нельзя переслать через границу процесса, поэтому `progress` у отдельной задачи под этим бэкендом запрещён — будет `ValueError`. Если нужен прогресс — используйте потоки.
    - **Дочерние процессы стартуют через `spawn`.** Что это значит для разработчика? На macOS и Windows интерпретатор Python в дочернем процессе запускается заново — и, следовательно, **заново импортирует ваш модуль**, чтобы получить код для воркера. Таким образом, ваш код не должен запускаться непосредственно в момент импорта — иначе импорт в дочернем процессе снова его запустит, и так по кругу. В обычном приложении проблема, как правило, не возникает: вы вызываете `extract_many(...)` только внутри какой-нибудь функции или метода, а не в момент импорта, и всё работает нормально. Однако, **в самозапускающихся скриптах**, где вызов пула стоит прямо в теле файла, он обязательно должен стоять под `if __name__ == "__main__":` — это позволяет избежать ловушки бесконечной рекурсии.
- **gevent** — если вы сделали `gevent.monkey.patch_all()`, `extract_many` это замечает и **бесшовно** переключается на `gevent.threadpool` (настоящие OS-потоки, которые хаб ждёт кооперативно). Никаких действий с вашей стороны не требуется — кооперативные потоки просто работают и не замораживают хаб.
- **Прогресс приходит из рабочего потока.** Колбэк `progress` (у `Archive.extract`, `AsyncArchive.extract` и у отдельного `ExtractJob`) вызывается из рабочего потока, а не из цикла событий. Обращаться из него к циклу событий можно только через `loop.call_soon_threadsafe`.

## Ограничения под Windows

- **Потоковое чтение `stream=True`** работает только для **источника-пути**. Для источника-`bytes` (или файлового объекта) будет `NotImplementedError`. Причина: такой источник необходимо обрабатывать через временный файл, а Windows не даёт удалить его, пока рабочий поток держит архив открытым. Используйте открытие архива через путь или `stream=False`. В целом, кейс редкий, поскольку stream=True используется для больших файлов, которые нежелательно помещать целиком в память (или они туда не поместятся сами). Передавать их, предварительно читая в память же, нет смысла, поэтому ограничение больше теоретическое, чем практическое.
- **`Backend.PROCESS`** на Windows работает через `spawn` — с теми же оговорками, что описаны выше в разделе [Параллелизм](#потоки-процессы-gevent--возможности-и-ограничения). Но есть ещё одна особенность: аргументы задач должны не иметь проблем при **сериализации через `pickle`**. При этом пути и `ExtractJob` сериализуются без проблем, так что такого рода код это не затрагивает.

Всё остальное — синхронный API, `AsyncArchive`, потоковый `extract_many` — работает под Windows так же, как под *nix.

## Исключения

Все ошибки, кроме `EntryNotFoundError`, наследуют `newtua.NewtuaError`:

| Исключение | Когда |
|---|---|
| `UnknownFormatError` | Формат архива не распознан. |
| `UnsupportedError` | Формат известен, но именно эта возможность не поддерживается. |
| `PasswordRequiredError` | Архив зашифрован, пароль не передан. |
| `WrongPasswordError` | Переданный пароль не подошёл. |
| `CorruptArchiveError` | Данные архива повреждены. |
| `MissingVolumeError` | Не хватает тома в многотомном архиве. |
| `UnsafePathError` | Путь записи попытался бы выйти за пределы папки распаковки. |

`EntryNotFoundError` наследует `KeyError`, а не `NewtuaError`, чтобы обращение по имени вело себя как обращение к словарю. Числовой индекс вне диапазона даёт `IndexError`, а операция с закрытым архивом — `ValueError`.

При `Backend.PROCESS` вы получаете ровно те же типизированные исключения, что и обычно: библиотека сама приводит ошибку движка к нужному классу ещё внутри процесса-воркера, поэтому до вас доезжает, например, `UnknownFormatError`, а не невнятная ошибка сериализации.

## Как тестировать

Мы позаботились, чтобы тесты были короткими:

- **Архив открывается прямо из байтов** — значит, временный файл в тесте не нужен.
- **Обработчик прогресса — это обычный список** (`events.append`), без каких-либо обёрток.
- **Исключения проверяются точным типом** через `pytest.raises`, а не разбором текста исключения.

```python
import pathlib, newtua, pytest

FIXTURES = pathlib.Path(__file__).parent / "fixtures"

def test_extracts_everything(tmp_path):
    data = (FIXTURES / "hello.7z").read_bytes()
    with newtua.Archive(data) as ar:
        report = ar.extract(tmp_path, wrapper=False)
    assert report.extracted == 1
    assert (tmp_path / "a.txt").read_text() == "hello 7z"

def test_collects_progress(tmp_path):
    events: list[newtua.ProgressEvent] = []
    with newtua.Archive(FIXTURES / "hello.7z") as ar:
        ar.extract(tmp_path, progress=events.append)
    assert any(isinstance(e, newtua.EntryStarted) for e in events)

def test_unknown_format(tmp_path):
    bad = tmp_path / "bad.bin"
    bad.write_bytes(b"это точно не архив")
    with pytest.raises(newtua.UnknownFormatError):
        len(newtua.Archive(bad))
```

`tmp_path` — обычная фикстура pytest, отдельная временная папка на каждый тест.

### Сборка колёс (wheels)

`abi3` (стабильный ABI с Python 3.11) — одно колесо на платформу покрывает все Python ≥ 3.11 для каждой платформы. Подсказки типов встроены: пакет типизирован в коде и помечен `py.typed`, а заглушка `_newtua.pyi` описывает скомпилированную часть.

```bash
maturin build --release
```

**Релизная сборка** описана в [`.github/workflows/wheels.yml`](.github/workflows/wheels.yml) и запускается по тегу `v*` или вручную. Она собирает колёса для Linux manylinux x86_64 + aarch64, macOS arm64 + x86_64 и Windows x86_64, плюс архив с исходниками (sdist). Затем каждое колесо ставится начисто, и по нему прогоняются тесты на Python 3.11 и 3.13: одно `abi3`-колесо должно покрывать обе версии — этот шаг и доказывает, что покрывает. Публикация на PyPI происходит только по тегу, через доверенную публикацию (trusted publishing), без API-токена.

Зависимости с C-кодом (bzip2, xz, AES для 7z, libunrar) собираются из исходников, поэтому сборочному образу нужен C-тулчейн — особенно это важно при кросс-сборке под aarch64. По той же причине воркфлоу выставляет `LZMA_API_STATIC=1`: без него `lzma-sys` подцепит ту liblzma, которая случайно оказалась на сборочной машине, и колесо потребует от пользователя библиотеку, которой у него может не быть.

> На самом новом CPython (например, >=3.14) сборка abi3-расширения может потребовать `PYO3_USE_ABI3_FORWARD_COMPATIBILITY=1`, пока PyO3 явно не перечислит этот интерпретатор в поддерживаемых.
